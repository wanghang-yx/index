#### 创建版本 
第一步：创建并切换到 v0.4 分支
首先，你需要基于当前的 master 分支创建一个新分支。

####  1. 确保你在 master 分支上（如果不在）
git checkout master

# 2. 创建并切换到 v0.4 分支
git checkout -b v0.4
解释：
checkout -b v0.4 是 git branch v0.4（创建分支）和 git checkout v0.4（切换分支）的合并简写。此时你的终端提示符应该会变成 (v0.4)。
第二步：修改代码
在 VS Code 或编辑器中打开 index.html，进行你想要的修改。例如：
将标题改为 <h3>我是 v0.4 版本</h3>。
添加一些新的 CSS 样式或内容。
保存文件。
第三步：提交更改到本地仓库
修改完成后，需要将更改记录到 Git 中。




# 1. 将所有修改添加到暂存区
git add .

# 2. 提交更改，并写上清晰的备注
git commit -m "更新到 v0.4 版本"
第四步：推送到 GitHub
这是最关键的一步。因为 v0.4 是一个新分支，GitHub 上还没有它，所以第一次推送时需要建立关联。




# 推送 v0.4 分支到 github 远程仓库
git push --set-upstream github v0.4
或者使用简写形式（仅限第一次推送该分支时）：




git push -u github v0.4
注意：
这里的 github 是你之前配置的远程仓库别名（从你之前的截图来看是这个）。如果你配置的是 origin，请把命令中的 github 换成 origin。
第五步：验证与后续操作
执行完上述命令后：
刷新 GitHub 网页：你会发现在分支下拉菜单中多了一个 v0.4 选项，且里面的代码是最新的。
关于合并：当 v0.4 开发完成并测试无误后，你可以随时切回 master 并将其合并进去：
bash

编辑



git checkout master
git merge v0.4
git push github master
通过以上步骤即可完成新版本的创建与推送。

然后合并到master
当你在 v0.4 分支上开发完成，并且已经成功推送到 GitHub 后，接下来就是将这个分支的代码合并回主分支 master，并同步到 GitHub。
请严格按照以下 4 个步骤 依次执行：
第一步：切回主分支
首先，你需要从 v0.4 分支切换回 master 分支，因为 Git 规定只能在主分支上接收合并。
bash

编辑



git checkout master
第二步：执行合并
将 v0.4 分支的代码合并到当前的 master 分支中。
bash

编辑



git merge v0.4
提示： 如果一切顺利，你会看到类似 Fast-forward（快进合并）或 Merge made by the 'recursive' strategy. 的成功提示。
第三步：推送到 GitHub
合并操作目前只发生在你自己的电脑上，最后一步必须把它推送到远程仓库，这样 GitHub 上的 master 才会更新。
bash

编辑



git push github master
第四步：（可选但推荐）清理本地分支
既然 v0.4 已经合并到 master 了，它的使命已经完成。为了保持本地分支列表的整洁，你可以把它删掉：
bash

编辑



git branch -d v0.4
💡 总结你的完整工作流
以后每次开发新版本，你的标准操作就是：
git checkout -b v0.x （创建并进入新版本分支）
修改代码...
git add . 和 git commit -m "..." （提交）
git push -u github v0.x （推送到 GitHub）
git checkout master 和 git merge v0.x （切回主分支并合并）
git push github master （把主分支也推送到 GitHub）
